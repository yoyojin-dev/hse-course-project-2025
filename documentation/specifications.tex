\subsubsection{Организация репозиториев}

Разработка системы ведётся в трёх отдельных репозиториях: бэкенд-репозитории, фронтенд-репозитории и инфраструктурном репозитории.

Бэкенд-репозиторий содержит серверную часть приложения: реализацию игровой логики и правил, серверный API, механизмы хранения и обработки состояния игровых сессий, управление пользователями и ролями, а также синхронизацию данных между клиентами в реальном времени.

Фронтенд-репозиторий содержит клиентскую часть приложения: пользовательский интерфейс для участников и ведущего, визуализацию состояния игры и метрик, а также логику взаимодействия с сервером через API.

Инфраструктурный репозиторий содержит конфигурационные файлы для развёртывания всей системы: конфигурацию веб-сервера Nginx, файлы Docker Compose для сборки локальных и production-окружений, скрипты деплоя и настройки CI/CD-пайплайна.

На начальном этапе проектирования рассматривалась классическая двухрепозиторная модель: один репозиторий для бэкенда и один для фронтенда. Это базовое решение не требует дополнительных усилий по организации и является распространённой практикой для небольших проектов.

По мере проектирования инфраструктуры возник вопрос о размещении файлов, не относящихся ни к логике бэкенда, ни к интерфейсу фронтенда: конфигурации Nginx, файлов Docker Compose, объединяющих оба сервиса, и скриптов деплоя. Рассматривались следующие варианты:

\begin{enumerate}
    \item \textbf{Разместить инфраструктурные файлы в бэкенд-репозитории}: отклонено, поскольку нарушает принцип единственной ответственности и смешивает бизнес-логику с инфраструктурными конфигурациями. Фронтенд-команда теряет прямой контроль над параметрами деплоя.
    \item \textbf{Разместить инфраструктурные файлы во фронтенд-репозитории}: отклонено по аналогичным причинам.
    \item \textbf{Выделить инфраструктурные файлы в отдельный репозиторий}: принято.
\end{enumerate}

Инфраструктурные конфигурации выделены в отдельный репозиторий.

\subsubsection{Стратегия управления ветвлением}
В ходе проектирования системы был проведен сравнительный анализ трех доминирующих стратегий управления ветвлением:

\begin{itemize}
    \item \textbf{Git Flow}~\cite{driessen-gitflow}: была отвергнута ввиду избыточной сложности для небольшой команды. Наличие долгоживущих веток develop и release создает дополнительные издержки и усложняет процесс интеграции готового кода в проект, что противоречит принципам минимизации потерь.
    \item \textbf{Trunk-Based Development}~\cite{trunk-based-dev}: метод предполагает интеграцию малых порций кода напрямую в master. Данный подход был признан рискованным на текущем этапе, так как он требует предельно высокого покрытия автоматизированными тестами и высокой дисциплины, отсутствие которых могло бы привести к дестабилизации основной ветки.
    \item \textbf{GitHub Flow}~\cite{github-flow}:данная модель наиболее близка к потребностям команды, однако в "чистом" виде она не обеспечивает необходимый уровень контроля над готовностью крупных функциональных блоков.
\end{itemize}

В результате была разработана авторская адаптация, которая основана на  GitHub Flow. Процесс строится на следующих правилах:

\begin{enumerate}
    \item Ветка master является абсолютно стабильной и всегда содержит код, готовый к эксплуатации. Любое слияние в master автоматически инициирует процесс деплоя на сервер.
    \item Для каждой крупной задачи создается отдельная ветка story/название-задачи. В отличие от GitHub Flow, где ветки создаются под мелкие фичи, наш подход предполагает накопление всех связанных с одной Стори изменений в одной ветке. Все технические задачи, декомпозированные на этапе аналитики, вливаются именно в эту ветку через локальные Pull Requests.
    \item Слияние ветки story в master происходит только тогда, когда все задачи данной стори выполнены, прошли код-ревью и комплексное тестирование. Это исключает ситуацию, когда в основную ветку попадает частично реализованный функционал.
    \item Команда следует принципу однопредметного потока из теории бережливого производства. Разработка новой Стори не начинается до момента полного завершения и слияния текущей в master.
\end{enumerate}

Такой подход полностью соответствует логике Закона Литтла~\cite{little-law}: ограничивая количество одновременно открытых веток крупных задач до одной, команда минимизирует объем незавершенной работы. Это позволяет сократить время прохождения задачи по циклу и повысить предсказуемость релизов. Такой подход к управлению ветвлением помогает поддерживать высокий уровень архитектурного качества и предотвращает накопление технического долга в основной ветке разработки.

\subsubsection{Технологический стек и автоматизация инфраструктуры}

Для реализации выбранной стратегии был применён набор технологий, который обеспечивает наилучшее соответствие между различными средами.

\begin{itemize}
    \item \textbf{Контейнеризация (Docker)}: Использование Docker позволяет инкапсулировать зависимости приложения в изолированные контейнеры. Это решает проблему несовместимости библиотек на разных машинах разработчиков.
    \item \textbf{Веб-сервер Nginx}: Выполняет роль обратного прокси-сервера на единственном сервере проекта, обеспечивая маршрутизацию трафика и безопасность.
\end{itemize}

Использование одного сервера на текущем этапе оправдано отсутствием необходимости в горизонтальном масштабировании при низкой начальной нагрузке. Это позволяет избежать "избыточной обработки" одного из видов потерь в Lean.

\subsubsection{Процесс релиза}

Релиз цикла фиксируется навешиванием Git-тега с семантической версией на последний коммит в ветке master. Это создает неизменяемую точку в истории проекта, к которой можно откатиться в случае обнаружения критических ошибок, обеспечивая устойчивость системы согласно принципам Agile.
