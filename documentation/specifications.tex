\section{Технические документации}

\subsection{Проектирование инфраструктуры и регламентация процессов управления исходным кодом}

Для обеспечения стабильного темпа разработки и минимизации технологических потерь была спроектирована специализированная инфраструктура и стратегия ветвления. В основу легли принципы изоляции компонентов, автоматизации рутинных операций и строгого ограничения незавершённого производства (WIP).

\subsubsection{Организация репозиториев}

Разработка системы ведётся в трёх отдельных репозиториях: бэкенд-репозитории, фронтенд-репозитории и инфраструктурном репозитории.

Бэкенд-репозиторий содержит серверную часть приложения: реализацию игровой логики и правил, серверный API, механизмы хранения и обработки состояния игровых сессий, управление пользователями и ролями, а также синхронизацию данных между клиентами в реальном времени.

Фронтенд-репозиторий содержит клиентскую часть приложения: пользовательский интерфейс для участников и ведущего, визуализацию состояния игры и метрик, а также логику взаимодействия с сервером через API.

Инфраструктурный репозиторий содержит конфигурационные файлы для развёртывания всей системы: конфигурацию веб-сервера Nginx, файлы Docker Compose для сборки локальных и production-окружений, скрипты деплоя и настройки CI/CD-пайплайна.

\paragraph{ADR-01: Выбор трёхрепозиторной структуры вместо двухрепозиторной}

\textbf{Контекст.} На начальном этапе проектирования рассматривалась классическая двухрепозиторная модель: один репозиторий для бэкенда и один для фронтенда. Это базовое решение не требует дополнительных усилий по организации и является распространённой практикой для небольших проектов.

\textbf{Проблема.} По мере проектирования инфраструктуры возник вопрос о размещении файлов, не относящихся ни к логике бэкенда, ни к интерфейсу фронтенда: конфигурации Nginx, файлов Docker Compose, объединяющих оба сервиса, и скриптов деплоя. Рассматривались следующие варианты:

\begin{enumerate}
    \item \textbf{Разместить инфраструктурные файлы в бэкенд-репозитории}: отклонено, поскольку нарушает принцип единственной ответственности и смешивает бизнес-логику с инфраструктурными конфигурациями. Фронтенд-команда теряет прямой контроль над параметрами деплоя.
    \item \textbf{Разместить инфраструктурные файлы во фронтенд-репозитории}: отклонено по аналогичным причинам.
    \item \textbf{Выделить инфраструктурные файлы в отдельный репозиторий}: принято.
\end{enumerate}

\textbf{Решение.} Инфраструктурные конфигурации выделены в отдельный репозиторий.

\textbf{Последствия.}
\begin{itemize}
    \item Изменения в инфраструктуре (например, обновление правил маршрутизации Nginx или добавление нового сервиса) производятся независимо от релизного цикла бэкенда и фронтенда.
    \item История изменений конфигурации деплоя чётко отделена от истории изменений продуктового кода.
    \item Инфраструктурные изменения могут проходить собственный цикл ревью и тестирования.
\end{itemize}

\subsubsection{Стратегия управления ветвлением}

В ходе проектирования системы был проведён сравнительный анализ трёх доминирующих стратегий управления ветвлением:

\begin{itemize}
    \item \textbf{Git Flow}~\cite{driessen-gitflow}: была отвергнута ввиду избыточной сложности для небольшой команды. Наличие долгоживущих веток \texttt{develop} и \texttt{release} создаёт дополнительные издержки и усложняет процесс интеграции готового кода в проект, что противоречит принципам минимизации потерь.
    \item \textbf{Trunk-Based Development}~\cite{trunk-based-dev}: метод предполагает интеграцию малых порций кода напрямую в \texttt{master}. Данный подход был признан рискованным на текущем этапе, так как требует предельно высокого покрытия автоматизированными тестами и высокого уровня дисциплины, отсутствие которых могло бы привести к дестабилизации основной ветки.
    \item \textbf{GitHub Flow}~\cite{github-flow}: данная модель наиболее близка к потребностям команды, однако в «чистом» виде она не обеспечивает необходимый уровень контроля над готовностью крупных функциональных блоков.
\end{itemize}

В результате была разработана авторская адаптация, которая интегрирует простоту GitHub Flow с жёстким контролем WIP-лимитов. Выбор данной стратегии обусловлен необходимостью гарантировать поставку только полностью завершённой ценности. Процесс строится на следующих правилах:

\begin{enumerate}
    \item Ветка \texttt{master} во всех трёх репозиториях (Frontend, Backend и Infrastructure) является абсолютно стабильной и всегда содержит код, готовый к эксплуатации. Любое слияние в \texttt{master} автоматически инициирует процесс деплоя на сервер.
    \item Для каждой крупной задачи создаётся отдельная ветка \texttt{story/название-задачи}. В отличие от GitHub Flow, где ветки создаются под мелкие фичи, данный подход предполагает накопление всех связанных с одной Стори изменений в одной ветке. Все атомарные технические задачи, декомпозированные на этапе аналитики, вливаются именно в эту ветку через локальные Pull Requests.
    \item Слияние ветки \texttt{story} в \texttt{master} происходит только тогда, когда все задачи данной Стори в рамках конкретного репозитория выполнены, прошли код-ревью и комплексное тестирование. Это исключает ситуацию, когда в основную ветку попадает частично реализованный функционал.
    \item Команда следует принципу однопредметного потока из теории бережливого производства. Разработка новой Стори не начинается до момента полного завершения и слияния текущей в \texttt{master}.
\end{enumerate}

Обоснование выбора данной модели базируется на Законе Литтла~\cite{little-law}: ограничивая количество одновременно открытых веток крупных задач до одной, команда минимизирует объём незавершённой работы. Это позволяет сократить время прохождения задачи по циклу и повысить предсказуемость релизов. Такой подход к управлению ветвлением помогает поддерживать высокий уровень архитектурного качества и предотвращает накопление технического долга в основной ветке разработки.

\subsubsection{Технологический стек и автоматизация инфраструктуры}

Для реализации выбранной стратегии был применён набор технологий, обеспечивающий наилучшее соответствие между различными средами.

\begin{itemize}
    \item \textbf{Контейнеризация (Docker)}: использование Docker позволяет инкапсулировать зависимости приложения в изолированные контейнеры. Это решает проблему несовместимости библиотек на разных машинах разработчиков.
    \item \textbf{Реестр образов (Container Registry)}: для координации между бэкенд- и фронтенд-разработчиками используется механизм публикации артефактов (образов).
    \item \textbf{Веб-сервер Nginx}: выполняет роль обратного прокси-сервера на единственном сервере проекта, обеспечивая маршрутизацию трафика и безопасность.
\end{itemize}

Использование одного сервера на текущем этапе оправдано отсутствием необходимости в горизонтальном масштабировании при низкой начальной нагрузке. Это позволяет избежать «избыточной обработки» — одного из видов потерь в Lean.

\paragraph{ADR-02: Обоснование применения Nginx как обязательного компонента инфраструктуры}

\textbf{Контекст.} В ходе проектирования архитектуры рассматривался вариант прямого доступа к контейнерам приложений без использования промежуточного слоя.

\textbf{Проблема.} Прямой доступ к контейнерам создаёт следующие ограничения:

\begin{itemize}
    \item \textbf{Нестандартные порты в URL:} пользователи обращались бы к сервису по адресу вида \texttt{http://ip:8080}, что не соответствует стандартам веб-сервисов.
    \item \textbf{Проблема CORS:} фронтенд (на одном порту) и бэкенд (на другом порту) воспринимаются браузером как разные источники, что блокирует запросы по политике Same-Origin. Решение этой проблемы на уровне кода приложения увеличивает его сложность.
    \item \textbf{Отсутствие HTTPS:} настройка SSL-сертификатов напрямую в контейнерах приложений значительно сложнее и менее надёжна, чем централизованное управление ими через прокси.
\end{itemize}

\textbf{Решение.} Nginx признан обязательным элементом инфраструктуры по следующим причинам:

\begin{enumerate}
    \item \textbf{Реализация паттерна «Фасад»:} Nginx скрывает внутреннюю структуру портов и сетевых адресов Docker-контейнеров, предоставляя клиенту единый интерфейс взаимодействия.
    \item \textbf{Минимизация CORS-рисков:} сведение фронтенда и бэкенда под единый домен и порт через проксирование устраняет проблемы CORS на уровне инфраструктуры, а не программного кода.
    \item \textbf{Снижение TCO (Total Cost of Ownership):} централизованное управление SSL-сертификатами и правилами перенаправления (например, с \texttt{http} на \texttt{https}) сокращает затраты на обслуживание системы.
\end{enumerate}

\textbf{Последствия.} Конфигурационные файлы Nginx хранятся в инфраструктурном репозитории, что позволяет управлять ими через Git и применять к ним те же практики ревью, что и к продуктовому коду. Таким образом, наличие Nginx является необходимым условием для перевода системы из статуса прототипа в статус отказоустойчивого веб-приложения~\cite{agile-practice-guide}.

\subsubsection{Оптимизация кросс-репозиторного взаимодействия}

Особое внимание уделено удобству разработки. Была проанализирована проблема зависимости разработчиков одной части системы от окружения другой. Рассмотренные варианты:

\begin{enumerate}
    \item \textbf{Полное дублирование}: разработчик клонирует все репозитории. Отвергнуто из-за сложности первоначальной настройки и когнитивной нагрузки.
    \item \textbf{Mock-серверы}: использование заглушек вместо реального сервиса. Отвергнуто из-за риска интеграционных ошибок и расхождения поведения заглушки с реальным кодом.
\end{enumerate}

Выбранное решение — разработка через Docker Registry. Для каждой из сторон процесс выглядит следующим образом:

\begin{enumerate}
    \item CI/CD-пайплайн бэкенда при каждом обновлении стабильной ветки автоматически собирает Docker-образ и публикует его в GitHub Container Registry с тегом \texttt{:latest}. Фронтенд-разработчик в своём локальном окружении выполняет единую команду \texttt{docker-compose up} и автоматически получает актуальный образ бэкенда без необходимости самостоятельно клонировать и запускать серверную часть.
    \item Симметричным образом CI/CD-пайплайн фронтенда собирает и публикует образ клиентской части в тот же реестр. Это позволяет бэкенд-разработчику поднять полное окружение с реальным фронтендом локально, не настраивая клиентскую часть вручную. Такой подход особенно ценен при интеграционном тестировании API.
\end{enumerate}

\subsubsection{Процесс релиза}

Релиз цикла фиксируется навешиванием Git-тега с семантической версией на последний коммит в ветке \texttt{master}. Это создаёт неизменяемую точку в истории проекта, к которой можно откатиться в случае обнаружения критических ошибок, обеспечивая устойчивость системы в соответствии с принципами управления конфигурациями~\cite{agile-practice-guide}.
