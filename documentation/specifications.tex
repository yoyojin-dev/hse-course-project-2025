\subsubsection{Организация репозиториев}

Разработка системы ведётся в трёх отдельных репозиториях: бэкенд-репозитории, фронтенд-репозитории и инфраструктурном репозитории.

Бэкенд-репозиторий содержит серверную часть приложения: реализацию игровой логики и правил, серверный API, механизмы хранения и обработки состояния игровых сессий, управление пользователями и ролями, а также синхронизацию данных между клиентами в реальном времени.

Фронтенд-репозиторий содержит клиентскую часть приложения: пользовательский интерфейс для участников и ведущего, визуализацию состояния игры и метрик, а также логику взаимодействия с сервером через API.

Инфраструктурный репозиторий содержит конфигурационные файлы для развёртывания всей системы: конфигурацию веб-сервера Nginx, файлы Docker Compose для сборки локальных и production-окружений, скрипты деплоя и настройки CI/CD-пайплайна.

На начальном этапе проектирования рассматривалась классическая двухрепозиторная модель: один репозиторий для бэкенда и один для фронтенда. Это базовое решение не требует дополнительных усилий по организации и является распространённой практикой для небольших проектов.

По мере проектирования инфраструктуры возник вопрос о размещении файлов, не относящихся ни к логике бэкенда, ни к интерфейсу фронтенда: конфигурации Nginx, файлов Docker Compose, объединяющих оба сервиса, и скриптов деплоя. Рассматривались следующие варианты:

\begin{enumerate}
    \item \textbf{Разместить инфраструктурные файлы в бэкенд-репозитории}: отклонено, поскольку нарушает принцип единственной ответственности и смешивает бизнес-логику с инфраструктурными конфигурациями. Фронтенд-команда теряет прямой контроль над параметрами деплоя.
    \item \textbf{Разместить инфраструктурные файлы во фронтенд-репозитории}: отклонено по аналогичным причинам.
    \item \textbf{Выделить инфраструктурные файлы в отдельный репозиторий}: принято.
\end{enumerate}

Инфраструктурные конфигурации выделены в отдельный репозиторий.

\subsubsection{Стратегия управления ветвлением}
Перед началом разработки был проведен сравнительный анализ трех стратегий управления ветвлением:

\begin{itemize}
    \item \textbf{Git Flow}~\cite{driessen-gitflow}: метод предполагает использование нескольких веток с долгим сроком жизни. Кроме ветки \texttt{main}, всегда содержащей стабильный код, существует \texttt{develop}, в которой ведётся разработка и интеграция проекта. Для каждой новой функциональности из неё создаётся ветка \texttt{feature/*}, которая после завершения вливается обратно в \texttt{develop}. Также в процессе участвуют ветки \texttt{release/*} (для финального тестирования) и \texttt{hotfix/*} (исправление критических ошибок). Эта стратегия была отвергнута ввиду избыточной сложности для небольшой команды. Наличие долгоживущих веток \texttt{develop} и \texttt{release} создает дополнительные издержки и усложняет процесс интеграции готового кода в проект, что противоречит принципам минимизации потерь.
    \item \textbf{Trunk-Based Development}~\cite{trunk-based-dev}: стратегия представляет из себя разработку с единственной основной веткой, в которую все разработчики постоянно интегрируют изменения. Этот подход минимизирует объём незавершённой работы. Все обновления вносятся очень маленькими порциями и проходят автоматическое тестирование. Для стабилизации перед выпуском может использоваться выделение короткоживущей релизной ветки. Это позволяет достичь высокой скорости интеграции и минимизировать конфликты слияния за счёт частоты и небольшого размера изменений. Данный метод был признан рискованным на текущем этапе, так как он требует предельно высокого покрытия автоматизированными тестами и высокой дисциплины, отсутствие которых могло бы привести к дестабилизации основной ветки.
    \item \textbf{GitHub Flow}~\cite{github-flow}: модель является более простой альтернативой Git Flow и строится на стабильной ветке \texttt{main} и временных тематических ветках (\texttt{feature-branch}), в которых ведётся вся разработка. После слияния с \texttt{main} обновления сразу попадают в продакшен. Данная стратегия удобна в использовании и наиболее близка к потребностям команды, однако в "чистом" виде она не обеспечивает необходимый уровень контроля над готовностью крупных функциональных блоков.
\end{itemize}

В результате была разработана авторская адаптация, которая основана на  GitHub Flow. Процесс строится на следующих правилах:

\begin{enumerate}
    \item Ветка \texttt{main} является абсолютно стабильной и всегда содержит код, готовый к эксплуатации. Любое слияние в неё автоматически инициирует процесс деплоя на сервер.
    \item Для каждой крупной задачи создается отдельная ветка \texttt{story/название-задачи}. В отличие от GitHub Flow, где ветки создаются под мелкие фичи, наш подход предполагает накопление всех изменений, связанных с одной стори (англ. story, пользовательская история — описание одной законченной функции продукта), в одной ветке. Все технические задачи, декомпозированные на этапе аналитики, вливаются именно в эту ветку через локальные Pull Requests.
    \item Слияние ветки \texttt{story} в \texttt{master} происходит только тогда, когда все задачи данной стори выполнены, прошли код-ревью и комплексное тестирование. Это исключает ситуацию, когда в основную ветку попадает частично реализованный функционал.
    \item Команда следует принципу однопредметного потока из теории бережливого производства. Разработка новой стори не начинается до момента полного завершения и слияния текущей в \texttt{master}.
\end{enumerate}

Такой подход полностью соответствует логике Закона Литтла~\cite{little-law}: ограничивая количество одновременно открытых веток крупных задач до одной, команда минимизирует объем незавершенной работы. Это позволяет сократить время прохождения задачи по циклу и повысить предсказуемость релизов. Такой подход к управлению ветвлением помогает поддерживать высокий уровень архитектурного качества и предотвращает накопление технического долга в основной ветке разработки.

\subsubsection{Технологический стек и автоматизация инфраструктуры}

Для реализации выбранной стратегии был применён набор технологий, который обеспечивает наилучшее соответствие между различными средами:

\begin{itemize}
    \item \textbf{Контейнеризация (Docker)}: Использование Docker позволяет собрать зависимости для проекта и все необходимые файлы для развертывания. Это решает проблему несовместимости библиотек на разных машинах разработчиков.
    \item \textbf{Веб-сервер Nginx}: Выполняет роль обратного прокси-сервера на единственном сервере проекта, обеспечивая маршрутизацию трафика и безопасность.
\end{itemize}

Использование одного сервера на текущем этапе оправдано отсутствием необходимости в горизонтальном масштабировании при низкой начальной нагрузке. Это позволяет избежать "избыточной обработки", одного из видов потерь в методологии Lean.

\subsubsection{Процесс релиза}

Релиз цикла фиксируется навешиванием Git-тега с семантической версией на последний коммит в ветке \texttt{master}. Это создает неизменяемую точку в истории проекта, к которой можно откатиться в случае обнаружения критических ошибок, обеспечивая устойчивость системы согласно принципам Agile.

\subsection{Страница лендинга}

В самом начале разработки командой было принято решение создать макет с правильной архитектурой, и в качестве результата второй стори был определен стек использованных технологий, а также выбрана и реализована страница лендинга.

\subsubsection{Сетевые маршруты и протоколы}

Сервис работает на двух независимых каналах связи:

\begin{itemize}
    \item \textbf{HTTP/1.1}: используется для передачи статических ресурсов клиентского приложения и выполнения операций, не требующих сохранения постоянного соединения (REST API).
    \item \textbf{WebSocket}: применяется для мгновенной двусторонней связи в реальном времени
    ~\cite{fette-websocket}, обеспечивая синхронизацию игрового состояния между пользователями с минимальной задержкой.
\end{itemize}

Было решено определиться с сетевыми протоколами на самом начальном этапе, так как в будущем предполагалась большая работа над Backend частью приложения.

\subsubsection{Архитектура клиентского приложения (Frontend)}

Было рассмотрено 2 варианта реализации архитектуры клиентского приложения: SPA (Single Page Application)~\cite{mesbah-spa} и MPA (Multi Page Application).

Было принято решение спроектировать приложение по паттерну Single Page Application (SPA)~\cite{mesbah-spa}. Данный подход минимизирует объем передаваемых по сети данных (после начальной загрузки передается только JSON-payload) и обеспечивает плавность пользовательского опыта, исключая перезагрузку страниц. От MPA было решено отказаться из-за большого недостатка в виде перезагрузки страницы при каждом обновлении состояния, что разрывало бы WebSocket соединение и приводило к постоянному прерыванию процесса игры.

Интерфейс построен с помощью библиотеки React~\cite{react-arch}. По маршруту \textbf{/} доступна страница лендинга приложения.

\subsubsection{Архитектура серверной части (Backend)}

Для серверной части проекта сначала был проведен сравнительный анализ языков программирования которые стоит использовать для написания. Команда выбирала между четырьмя языками программирования:
\begin{itemize}
\item \textbf{Python} — был первым рассматриваемым вариантом, так как каждый человек в команде имел много опыта программирования на нём. Несмотря на то, что с этим языком программирования работали все, и он позволяет быстро создать набросок серверной части, Python имеет ряд недостатков, среди которых: GIL (Global Interpreter Lock), который усложнит поддержку синхронизации множества игроков, более низкая производительность по сравнению с компилируемыми языками, большое потребление оперативной памяти и высокая зависимость от сторонних библиотек. Поэтому были рассмотрены другие варианты.~\cite{high-perf-python}
\item \textbf{Java} — Довольно популярный выбор в связи с высокой производительностью, но такая производительность была бы для нашего проекта избыточна. В то время как JVM требует много памяти и долго стартует, пользователи нашего приложения практически не заметят разницы по сравнению с другими кандидатами. Также этот язык из всех людей в команде знал один человек, что являлось для нас также недопустимым, ведь работу над Backend частью мы хотели разделить на несколько человек.
\item \textbf{PHP} — Вариант был предложен в связи с его популярностью для написания веб-приложений. Однако изучив тонкости языка было обнаружено, что в PHP отсутствует нативная поддержка WebSocket соединений, что было критическим недостатком и сразу поставило низкий приоритет выбора этого языка программирования.~\cite{PHP-book}
\item \textbf{Golang} — Самый подходящий для нас вариант. Язык позволяет легко скомпилировать приложение в один бинарный файл без зависимостей. Горутины и каналы позволяют держать десятки тысяч WebSocket-соединений с минимальным расходом RAM. Также несколько участников команды знали этот язык программирования, что дало возможность распределить нагрузку по написанию серверной части проекта на несколько участников команды.~\cite{donovan-go}
\end{itemize}

Таким образом, было принято решение реализовать серверную часть на языке Go.

Для MVP версии проекта было решено хранить состояние игры в оперативной памяти сервера. Благодаря выбору Go, где каждое WebSocket-соединение потребляет минимальное количество RAM, такой подход оказался оправданным даже при большом количестве одновременных подключений. Также в Go было легко избежать проблем с рассинхронизацией при одновременном действии игроков с помощью мьютексов (Mutex), которые защищают общее состояние игры от race condition — ситуации, когда несколько горутин пытаются одновременно изменить одни и те же данные.

\subsubsection{Синхронизация состояния: WebSocket и механизмы отказоустойчивости}

Для синхронизации между клиентами была выбрана архитектура Event
-
Driven Architecture через протокол WebSocket.
При подключении к эндпоинту \texttt{/ws/game} клиент инициализирует постоянное соединение. 
Сервер регистрирует соединение в памяти RAM с привязкой к идентификатору сессии.
При любом обращении к REST API и изменении состояния игры (например, эндпоинты \texttt{/move}, \texttt{/skip\_turn}), сервер инициирует broadcast-рассылку нового состояния всем активным подписчикам данной сессии~\cite{pimentel-websocket}.

Для обеспечения отказоустойчивости на клиентской стороне реализован паттерн Graceful Degradation. В случае недоступности WebSocket-соединения, клиент автоматически переходит на механизм периодического опроса (HTTP Polling) эндпоинта \texttt{/api/game/:code}. При восстановлении websocket соединения polling деактивируется.

\subsection{Архитектура и логика страницы игровой доски (Kanban)}

Реализация интерфейса и серверной части работы игровой доски разделена на несколько уровней абстракции, обеспечивая высокую степень изоляции компонентов и мгновенную синхронизацию состояния.

\subsubsection{Клиентская архитектура (Frontend)}

Пользовательский интерфейс игровой страницы реализован как функциональный компонент на базе библиотеки \textbf{React}~\cite{react-arch}, с TypeScript. Для эффективного рендеринга и контроля жизненного цикла локальных данных применяются стандартные React-хуки:
\begin{itemize}
    \item \texttt{useState} — для хранения локальных флагов (статусы загрузки, всплывающие уведомления).
    \item \texttt{useCallback} — для кеширования обработчиков событий (таких как загрузка состояния), защищая дочерние компоненты от лишних обновлений.
    \item \texttt{useRef} — для работы с Drag-and-Drop операциями и хранения промежуточного состояния перемещаемой карточки без вызова повторного рендеринга.
\end{itemize}

Чтобы избежать рассинхронизации данных был использован паттерн в котором единственный источник правды - это payload от сервера. При совершении действия (например, дропе карточки) локальное состояние не мутирует мгновенно — вместо этого отправляется WebSocket запрос к серверу, который валидирует действие по правилам игры. Только после получения подтвержденного нового состояния от сервера, React перерисовывает доску.

\subsubsection{Структура игрового пространства}

Игровая доска разделена на два иерархических уровня отображения:

\textbf{Проектный уровень}
Отображает глобальный прогресс команд и разделен на три макро-колонки. Структура данных проектов опирается на тип \texttt{ProjectCard} на сервере:
\begin{itemize}
    \item \textbf{Сделать} — список доступных проектов. Система динамически подсчитывает количество задач (прогресс: \texttt{done\_tasks} / \texttt{total\_tasks}).
    \item \textbf{Интеграция} — проекты, в которых часть команд закрыла все свои задачи. Для этого этапа реализован счетчик игровых дней в интеграции (поле \texttt{days\_in\_integration}) и визуальная система отображения начисленных штрафов (\texttt{penalty\_issued}).
    \item \textbf{Готово} — завершенные проекты с фиксацией дня завершения (\texttt{done\_day}).
\end{itemize}

\textbf{Уровень задач команд}
Для каждой команды формируется собственная Kanban-доска задач, отражающая текущий рабочий процесс. Эта доска основывается на мапе состояний на сервере (задается массивом \texttt{boardStages = []string\{"ready", "in\_progress", "review", "done"\}}) и строго лимитирована по колонкам:
\begin{itemize}
    \item \texttt{ready} (К выполнению)
    \item \texttt{in\_progress} (В работе)
    \item \texttt{review} (Ревью)
    \item \texttt{done} (Готово)
\end{itemize}
Разделение на два уровня позволяет одновременно видеть как высокоуровневую картину продукта для фасилитатора, так и микроменеджмент ресурсов внутри команд.

\subsubsection{Распределенное состояние и отказоустойчивость синхронизации}

\textbf{Единый источник истины (Single Source of Truth) и блокировки}

Весь актуальный стейт хранится в оперативной памяти бэкенд сервера (в кастомной In-Memory реализации интерфейса \texttt{Storage}). Для предотвращения состояний гонки сервер инкапсулирует каждое обращение к данным при помощи примитивов синхронизации: стандартных \textbf{\texttt{sync.RWMutex}} и \textbf{\texttt{sync.Mutex}}. Таким образом, любое действие, меняющее расположение карточки на доске накладывает системную блокировку структуры данных на уровне сервера. 

После применения валидированного изменения сервер генерирует целостное обновленное состояние и посредством broadcast-рассылки на \textbf{WebSockets} пересылает новый стейт всем активным клиентам данной сессии, гарантируя строгую согласованность данных.

\subsection{Игровой цикл и механика непрерывного улучшения (Continuous Improvement)}

Бизнес-логика приложения смоделирована на основе пошаговой симуляции работы команды разработки продукта. Архитектура игрового процесса реализует паттерны Agile-методологий и разбита на несколько строго контролируемых фаз состояний, за переходы между которыми отвечает фасилитатор. Каждая смена фазы сопровождается логгированием событий (сохранением истории) и мгновенным оповещением всех активных клиентов по WebSocket.

\subsubsection{Инициализация и запуск}

Игровой цикл начинается с состояния \texttt{setup}. На этом этапе ведущий инициализирует сессию с заданным максимальным количеством игровых дней (\texttt{max\_days}), а игроки присоединяются к выбранным командам (REST запрос \texttt{join}). Далее фасилитатор выбирает стартовые проекты (\texttt{start\_project}), формируя первоначальный бэклог продукта.

После завершения подготовительных действий сервер переводит игру в состояние \texttt{running}, и запускает первый игровой день.

\subsubsection{Ежедневная продуктовая разработка}

В активной фазе (\texttt{running}) каждый игровой день имитирует рабочий спринт, в течение которого разработчики тянут задачи из бэклога и передвигают их по своей Kanban-доске, применяя Drag-and-Drop интерфейс. Вся логика перемещения защищена на сервере строгими валидаторами:
\begin{itemize}
    \item \textbf{Проверка прав:} перемещать задачи может только участник, прикрепленный к конкретной команде, предотвращая кросс-командные коллизии хода.
    \item \textbf{Бросок монеты:} В начале каждого дня сервер генерирует случайное событие для каждого игрока с помощью \texttt{rand}. Подобная геймификация имитирует ограничения рабочего дня программиста.
    \item \textbf{Контроль лимитов:} Ключевым ограничением выступают WIP-лимиты команды, встроенные структуру состояния на сервере. При попытке перетащить карточку в заполненную колонку (например, \texttt{in\_progress}), сервер выбросит исключение и вернет клиенту код ошибки \texttt{409 Conflict} — "достигнут лимит WIP".
\end{itemize}
Завершение рабочего дня инициируется фасилитатором. Сервер инкрементирует счетчики времени в интеграции для зависимых проектов и сбрасывает статус ходов игроков.

\subsubsection{Ретроспектива и подстройка процесса}

Для внедрения в игру практики Continuous Improvement, сервер содержит триггер: каждое прохождение пяти циклов/дней (\texttt{(CurrentDay-1) \% 5 == 0}) автоматически переводит игру в состояние \textbf{Ретроспективы} (\texttt{Phase = "retro"}).

Данная фаза останавливает все возможности для перемещения тасок по доске, вынуждая команды анализировать метрики производительности и находить узкие горлышки в своем процессе разработки. Только в фазе ретроспективы команды вместе с ведущим имеют право адаптировать процесс под свои реалии.

Через эндпоинт \texttt{set\_wip} фасилитатор может увеличить или уменьшить \texttt{WIP-лимит} на Kanban-доске конкретной команды. Это действие логируется в глобальную историю игры. По завершении обсуждения, ведущий возобновляет работу (\texttt{continue}), и игра возвращается в фазу \texttt{running}.

\subsubsection{Завершение игры}

Игровой цикл заканчивается по достижении установленного на старте лимита времени. Игра переходит в финальную фазу. На этом этапе доска блокируется, а команды могут проводить финальный анализ: подсчет выполненных проектов, затраченных циклов и количества штрафов. Этот процесс наглядно демонстрирует влияние лимитов незавершенной работы на общую пропускную способность команд разработки.
